### 📋 ПОСТАНОВКА ЗАДАЧИ

**Название:** Веб-версия Brainify с авторизацией через Telegram и гостевым режимом

**Цель:** Расширить канал привлечения пользователей за пределы Telegram Mini App. Дать возможность заходить на brainify.ellow.tech из обычного браузера, пробовать продукт без входа (снижение барьера), авторизовываться через Telegram и получать полный доступ с проверкой подписки. Конечная цель — увеличить acquisition, конверсию в активных пользователей и конверсию в платную подписку.

**Проблема:** Сейчас Brainify работает **исключительно** внутри Telegram Mini App. Вся авторизация завязана на `initData` из Telegram WebApp SDK (HMAC-SHA256). Вся фронтенд-логика (`useAuth`, `api/client.ts`, `lib/telegram.ts`) получает `initData` через `window.Telegram.WebApp`. Если пользователь заходит в браузере напрямую — видит ошибку авторизации, продукт недоступен. Это означает:
- невозможно привлекать пользователей через SEO, директ-ссылки, соцсети, лендинги;
- невозможно дать «попробовать» продукт без установки Telegram;
- потеря потенциальных пользователей, которые не готовы сразу входить через Telegram.

**Контекст:**
- Стек: Preact + Vite (webapp), Fastify (API), Drizzle ORM + PostgreSQL, grammY (bot), Redis, Docker Compose, Nginx
- Монорепо: `packages/api`, `packages/bot`, `packages/webapp`, `packages/shared`
- Авторизация: единственный способ — `initData` через Telegram WebApp SDK, middleware `authMiddleware` валидирует HMAC-SHA256
- Пользователи идентифицируются по `telegramId` (bigint, unique) в таблице `users`
- Модель подписки: trial (7 дней) → free (3 задания/день, без PvP, без графиков) → active (полный доступ, Telegram Payments / Stars)
- Домен: `brainify.ellow.tech`, Nginx reverse proxy, HTTPS
- CSP сейчас: `frame-ancestors 'self' https://web.telegram.org https://*.telegram.org`
- API клиент (`api/client.ts`) отправляет `Authorization: tma <initData>` с каждым запросом
- При первом запросе `/api/user/me` автоматически создаёт пользователя по `telegramId`

### 🎯 ГРАНИЦЫ (SCOPE)

**Входит:**

1. **Гостевой режим (без входа)** — урезанная/пробная версия продукта, доступная без авторизации:
   - определить, какие функции доступны гостю (каталог заданий, ограниченное количество попыток)
   - API-эндпоинты, не требующие авторизации (или работающие с анонимной сессией)
   - фронтенд: отображение ограниченного функционала, призыв к авторизации

2. **Авторизация через Telegram Login Widget на вебе** — полноценный вход:
   - интеграция Telegram Login Widget (кнопка «Войти через Telegram» на веб-странице)
   - серверная валидация данных от Telegram Login Widget (отличается от initData Mini App)
   - создание серверной сессии / выдача JWT после успешной авторизации
   - связь web-пользователя с существующим аккаунтом по `telegramId`

3. **Проверка подписки после входа** — для авторизованных:
   - если подписка active/trial → полный доступ
   - если подписка free → предложить подписаться (paywall)
   - UI для отображения статуса подписки и CTA на подписку

4. **Совместимость с текущим Mini App** — существующая авторизация через `initData` должна продолжать работать без изменений

5. **Адаптация веб-приложения** — webapp должен работать и в Telegram, и в обычном браузере:
   - определение среды (Telegram Mini App vs обычный браузер)
   - роутинг и навигация, адаптированные под обе среды
   - UI-адаптация (цветовая тема Telegram vs собственная тема для веба)

6. **Адаптация API** — поддержка двух способов авторизации:
   - `Authorization: tma <initData>` — для Mini App (как сейчас)
   - `Authorization: Bearer <jwt>` — для веб-сессии после Telegram Login
   - опциональная авторизация для гостевых эндпоинтов

7. **Оплата подписки из веб-версии** — механизм, позволяющий пользователю оплатить подписку не из Telegram (редирект в бот или внешний платёжный провайдер)

**Не входит:**

- Регистрация через email / пароль / Google / Apple (только Telegram)
- Мобильное нативное приложение
- Отдельный лендинг / маркетинговый сайт (только сам продукт)
- Миграция с Preact на другой фреймворк
- Изменение существующей логики генерации заданий (shared)
- Изменение бота (bot), кроме возможной генерации deep link для подписки
- Переработка дизайна всего приложения (только минимальная адаптация под среду)

### ⚠️ ОГРАНИЧЕНИЯ

- **Обратная совместимость:** текущая Mini App авторизация (`initData`) должна работать без изменений; существующие пользователи не должны быть затронуты
- **Telegram Login Widget** требует настройки домена в BotFather (`/setdomain`) — нужен доступ к BotFather
- **Один пользователь = один Telegram ID:** и Mini App, и веб-авторизация привязываются к одному `telegramId`, данные общие
- **Оплата Telegram Payments / Stars работает только внутри Telegram:** для веб-версии нужен альтернативный способ подписки или редирект в бот
- **Бюджет бандла webapp:** ≤ 200 КБ gzip (добавление JWT-логики, Telegram Login Widget, гостевого режима не должно критично раздуть бандл)
- **Безопасность:** JWT должен быть подписан серверным секретом; Telegram Login данные должны валидироваться через HMAC-SHA256 (аналогично initData, но по другому алгоритму); гостевой режим не должен давать доступ к чужим данным
- **Стек:** Node.js ≥ 20, TypeScript, Fastify, Preact, Drizzle ORM — изменения в рамках существующего стека

### 🔍 ФАКТЫ И ПРЕДПОЛОЖЕНИЯ

**Факты** (подтверждённое):

- Авторизация сейчас — только через Telegram `initData` с HMAC-SHA256 (`packages/api/src/auth/middleware.ts`)
- Таблица `users` идентифицирует пользователей по `telegramId` (bigint, unique)
- API клиент (`packages/webapp/src/api/client.ts`) берёт `initData` из `window.Telegram.WebApp.initData` и отправляет как `Authorization: tma <initData>`
- `useAuth` хук вызывает `/api/user/me`, который создаёт пользователя при первом запросе
- Модель подписки: trial (7 дней) → free → active; проверка в `services/subscription.ts`
- Оплата работает через Telegram Payments / Stars (обработка в `packages/bot/src/payments/`)
- CSP в nginx: `frame-ancestors 'self' https://web.telegram.org https://*.telegram.org`
- Домен: `brainify.ellow.tech`
- Free-пользователи: 3 задания/день, без PvP, без детальных графиков
- Webapp определяет Telegram-среду через `window.Telegram?.WebApp`
- Nginx проксирует `/api/` → `api:3000`, статика webapp раздаётся напрямую

**Предположения** (требует проверки):

- Telegram Login Widget подойдёт для авторизации на вебе (предполагаем, что Telegram разрешает настроить domain для бота)
- Гостевой режим = доступ к каталогу заданий + ограниченное число попыток (без сохранения прогресса, без соцфункций) — **нужно подтвердить scope гостевого доступа**
- JWT-сессия на вебе будет достаточно безопасной альтернативой `initData` (время жизни, refresh, хранение)
- Для оплаты из веб-версии достаточно редиректа в Telegram-бот (deep link) — **нужно подтвердить, приемлем ли UX**
- Данные гостя (попытки без входа) не нужно сохранять и переносить после входа — **нужно подтвердить**

### 📌 ДОПУЩЕНИЯ

- Telegram Login Widget — основной и единственный способ авторизации на вебе (других провайдеров не будет на данном этапе)
- Гостевой режим = ограниченный доступ без сохранения данных, цель — дать попробовать и конвертировать в авторизацию
- Один пользователь = один Telegram ID, независимо от точки входа (Mini App или веб)
- Для оплаты из веб-версии на первом этапе используем deep link в Telegram-бот
- Данные гостевых сессий не сохраняются серверно и не переносятся после входа
- Структура БД (schema) не требует изменения таблицы `users` (telegramId остаётся основным идентификатором)
- Webapp остаётся единым приложением (Preact), работающим в двух средах (Telegram + браузер), а не два отдельных приложения

### ❓ ОТКРЫТЫЕ ВОПРОСЫ

1. **Какой именно функционал доступен гостю?**
   - Рабочая гипотеза: каталог заданий (просмотр), 1-3 попытки без сохранения прогресса, без лидерборда, дуэлей, друзей, достижений, профиля. После лимита — экран «Войдите через Telegram для полного доступа».

2. **Нужно ли сохранять данные гостевых сессий и переносить их после входа?**
   - Рабочая гипотеза: нет, гостевые попытки — демо, результаты не сохраняются. Это упрощает реализацию и мотивирует войти.

3. **Как оплачивать подписку из веб-версии?**
   - Рабочая гипотеза: deep link в бота (`t.me/BrainifyBot?start=subscribe`). На первом этапе — достаточно. В будущем — внешний платёжный провайдер (Stripe / ЮKassa).

4. **Нужна ли отдельная тема/дизайн для веб-версии (вне Telegram)?**
   - Рабочая гипотеза: минимальная адаптация — собственная дефолтная тема для браузера (сейчас CSS-переменные берутся из Telegram theme), без полного редизайна.

5. **Нужно ли менять CSP в nginx для веб-режима?**
   - Рабочая гипотеза: да, `frame-ancestors` нужно расширить или убрать ограничение фреймов для веб-доступа; добавить домен Telegram Login Widget в `script-src`.

6. **Как обрабатывать ситуацию, когда пользователь вошёл на вебе, но у него нет аккаунта (не пользовался ботом)?**
   - Рабочая гипотеза: создавать аккаунт при первом входе через Telegram Login (аналогично текущей логике `/api/user/me` для Mini App). Пользователь получает trial.

### 📦 ГОТОВО ДЛЯ ПЛАНИРОВЩИКА

**Тип задачи:** новая функциональность (с элементами рефакторинга auth-слоя)

**Приоритет (оценка):** высокий — расширяет канал привлечения, снижает барьер входа, увеличивает потенциальный acquisition

**Зависимости:**
- Настройка домена в BotFather для Telegram Login Widget
- Текущая auth-система должна остаться работоспособной
- Модель подписки и проверка лимитов (subscription service) — используется как есть, расширяется для гостевого режима

**Критерий готовности:**
1. Пользователь заходит на `brainify.ellow.tech` из обычного браузера и видит рабочий интерфейс (не ошибку авторизации)
2. Без входа доступен гостевой режим: каталог и ограниченное число заданий
3. Кнопка «Войти через Telegram» работает, авторизует, создаёт сессию
4. После входа — подписка проверяется, отображается текущий статус (trial/free/active)
5. Если подписка free — предлагается подписаться (CTA с переходом в бот)
6. Существующая Mini App авторизация (`initData`) работает без изменений
7. Оба варианта входа (Mini App и веб) привязаны к одному аккаунту по `telegramId`

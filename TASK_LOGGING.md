### 📋 ПОСТАНОВКА ЗАДАЧИ

**Название:** Добавить диагностические логи и устранить нестабильность TMA

**Цель:** Обеспечить предсказуемый запуск Telegram Mini App и видимость ошибок в реальном времени — чтобы при любом сбое сразу было понятно, что пошло не так и где.

**Проблема:** Две связанные проблемы:
1. **Нестабильный запуск TMA** — Mini App иногда показывает «An error occurred», а иногда загружается. Причина не очевидна: может быть race condition при создании пользователя, протухший initData, сбой миграции, отсутствие таблиц, 500 от API — но пользователь видит только заглушку «An error occurred» без деталей.
2. **Пустой каталог заданий** — после загрузки TMA на экране нет заданий. Причина: таблица `tasks` пустая — миграция создаёт структуру (DDL), но не наполняет данными (нет seed). В `taskRegistry` определены 7 типов задач (schulte, sequence_memory, arithmetic, stroop, number_series, memory_pairs, pattern_search), но в БД нет ни одной записи.

**Контекст:**
- Стек: Preact (Mini App) → nginx reverse proxy → Fastify API → PostgreSQL + Redis
- Авторизация: Telegram initData → HMAC validation → создание пользователя при первом визите
- Миграции: скрипт `migrate.ts` (drizzle-orm/migrator) запускается перед стартом API через CMD в Dockerfile
- Текущее логирование: Fastify default logger (`{ logger: true }`) — логирует запросы, но не бизнес-контекст ошибок
- Фронтенд: `useAuth()` ловит ошибку с `getMe()` → выставляет `error` → App отображает `t('common.error')` — информация потеряна, статус-код и сообщение не показываются
- В `TaskCatalog` ошибка загрузки `.catch(() => setTasks([]))` молча скрывается — задания просто пустые

### 🎯 ГРАНИЦЫ (SCOPE)

**Входит:**
- Добавить seed-данные для таблицы `tasks` — 7 заданий, соответствующих `taskRegistry`
- Улучшить отображение ошибок в Mini App — вместо глухого «An error occurred» показывать категорию ошибки (сеть, авторизация, сервер) и HTTP статус
- Добавить `console.error` / `console.warn` в ключевых точках фронтенда:
  - `useAuth.ts` — какая ошибка при `getMe()` (статус, message)
  - `api/client.ts` — при fetch failure (network error vs HTTP error)  
  - `TaskCatalog.tsx` — при ошибке загрузки заданий
  - `app.tsx` — при ошибке рендера (error boundary)
  - `lib/telegram.ts` — инициализация WebApp (есть ли initData, есть ли user)
- Добавить более детальные логи на API:
  - Логировать факт и причину отказа в auth middleware (missing header, invalid initData, expired)
  - Логировать результат миграции (успех / ошибка / уже применено)
  - Логировать ошибки в роутах users и tasks с контекстом

**Не входит:**
- Перестройка системы логирования (structured logging, transport в файлы, log aggregation)
- Внедрение мониторинга или алертинга
- Изменение бизнес-логики задач
- Изменение архитектуры авторизации
- Unit-тестирование

### ⚠️ ОГРАНИЧЕНИЯ

- Продакшн-окружение — Docker-контейнеры, логи доступны через `docker logs`
- `drizzle-kit` недоступен в рантайме (devDependency) — seed нужно реализовать через `drizzle-orm` программно
- Фронтенд — Preact (не React), нет встроенного ErrorBoundary, нужно реализовать через компонент
- initData от Telegram может протухать (24ч), нужно корректно обрабатывать этот случай
- Mini App открывается в контексте Telegram — `console.log` не виден напрямую, нужно или показывать в UI, или полагаться на серверные логи

### 🔍 ФАКТЫ И ПРЕДПОЛОЖЕНИЯ

**Факты** (подтверждённое):
- Миграция `0000_past_ultimates.sql` создаёт все таблицы, но НЕ содержит INSERT для tasks
- `taskRegistry` содержит 7 типов задач: schulte, sequence_memory, arithmetic, stroop, number_series, memory_pairs, pattern_search
- API логирует запросы через Fastify logger, но без бизнес-контекста (не логирует, почему отклонена авторизация)
- Фронтенд отображает `t('common.error')` при ошибке — без деталей
- `TaskCatalog` молча проглатывает ошибки загрузки (`.catch(() => setTasks([]))`)
- `api/client.ts` бросает `ApiError(status, message)`, но вызывающий код теряет детали
- Auth middleware возвращает разные 401 ответы (`Missing authorization header`, `Invalid initData`), но не логирует причину
- migrate.ts использует `drizzle-orm/postgres-js/migrator` — будет работать в рантайме без drizzle-kit

**Предположения** (требует проверки):
- Нестабильный запуск TMA связан с тем, что API возвращает 500 (таблицы не созданы) или с race condition миграции vs первый запрос
- После добавления seed таблица tasks будет содержать нужные данные и каталог заполнится
- Ошибка «An error occurred» — это именно `t('common.error')` из app.tsx, а не ошибка самого Telegram WebApp SDK

### 🚧 РИСКИ

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Seed дублирует данные при повторном запуске API | Высокая | Среднее | Использовать `onConflictDoNothing` при вставке |
| Миграция не успевает завершиться до первого запроса | Средняя | Высокое | migrate.ts выполняется до старта Fastify (CMD sequential) — но нужно проверить ожидание health check |
| console.error на фронтенде недоступен пользователю в Telegram | Средняя | Среднее | Дублировать критичные ошибки в UI + полагаться на серверные логи |
| Seed-данные рассинхронизируются с taskRegistry при добавлении новых задач | Низкая | Среднее | Генерировать seed из taskRegistry, а не хардкодить |

### 📌 ДОПУЩЕНИЯ

- **Точная маппинг type→category для каждой задачи** — гипотеза: schulte→attention, sequence_memory→memory, arithmetic→logic, stroop→attention, number_series→logic, memory_pairs→memory, pattern_search→attention. Нужно проверить по коду shared-пакета.
- **Куда именно смотреть ошибку фронтенда, если TMA открыта в Telegram?** — гипотеза: серверные логи (docker logs brainify-api) покрывают 90% случаев, для фронта — показывать в UI.
- **Нужно ли seed данные для других таблиц?** — гипотеза: нет, users/attempts и прочие создаются динамически при использовании.

### 📦 ГОТОВО ДЛЯ ПЛАНИРОВЩИКА

**Тип задачи:** исправление дефекта + улучшение наблюдаемости

**Приоритет (оценка):** критический (платформа не функциональна без seed данных и непредсказуемо падает без диагностики)

**Зависимости:**
- Миграции должны быть уже применены (таблицы существуют) — решено в предыдущей задаче (migrate.ts в CMD)
- `taskRegistry` в packages/shared — источник истины для типов задач

**Критерий готовности:**
1. После запуска API таблица `tasks` содержит 7 записей (по одной на каждый тип из taskRegistry)
2. При открытии TMA отображается каталог с заданиями (не пустой список)
3. При ошибке загрузки TMA — в UI виден конкретный тип ошибки (401/500/network) вместо «An error occurred»
4. В логах API (`docker logs brainify-api`) видны причины отказа авторизации, ошибки роутов и результат seed
5. Повторный запуск API не создаёт дубликаты в tasks (идемпотентный seed)
